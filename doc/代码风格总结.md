# 我的嵌入式代码风格总结

本文档旨在总结我在嵌入式开发中形成的个人代码风格。这不是关于格式化或代码规范的传统讨论，而是关于如何组织思维、表达意图、构建可维护嵌入式系统的个人方法论。

## 一、代码分区与组织

### 1.1 主文件的区块划分

在我的项目中，主文件（如main.c）采用了严格的区块划分方式。我将代码划分为以下几个明确的区域：

```
//Include Begin
//Include End

//define Begin
//define End

//variables Begin
//variables End 

//Handle Begin
//Handle End
```

这种划分方式的核心思想是**关注点分离**。在嵌入式开发中，我们需要同时处理硬件配置、数据定义、外设句柄等多个维度。将这些不同性质的内容物理分隔，可以让我们在阅读代码时快速定位到目标区域。

特别值得注意的是variables区域的结尾处有一个**多余的空格**（`//variables End `），这是我在长期实践中形成的一个小习惯——用空格来区分"End"标记和注释内容的结束，使注释视觉上更柔和。

### 1.2 main函数的阶段化结构

在我所有的main函数中，都遵循着相似的启动流程结构：

```c
int main()
{
    //System Initialize Begin
    delay_init_config();
    nvic_priority_group_set(NVIC_PRIGROUP_PRE2_SUB2);
    //System Initialize End
    
    //Configuration Begin
    USB_UART=(USART){USART0, GPIOA, GPIO_PIN_9, GPIOA, GPIO_PIN_10, 115200, USART_PM_NONE, USART_WL_8BIT, USART_STB_1BIT};
    //Configuration Ends
    
    //Peripherals Initialize Begin
    usart_init_config(&USB_UART);
    //Peripherals Initialize End
    
    //Other Initialize Begin
    //Other Initialize End
    
    //User Begin
    //User End
    
    while(1)
    {
        // TODO
    }
    return 0;	
}
```

这个结构体现了嵌入式开发的**认知模型**：
- **System Initialize**：与硬件核心相关的初始化，通常只执行一次
- **Configuration**：定义外设的配置参数，这通常是数据声明而非动作
- **Peripherals Initialize**：真正执行外设的初始化动作
- **Other Initialize**：上述分类无法涵盖的初始化内容
- **User**：用户代码区域，即主要业务逻辑所在

这种分层的核心理念是**延迟执行**。配置数据声明和实际初始化动作是分开的，这让我可以在启动早期集中定义所有参数，之后再逐个初始化。

## 二、命名体系的哲学

### 2.1 前缀约定

我在项目中建立了一套一致的前缀约定，这来自于对嵌入式开发特殊性的理解：

- **`h`前缀**：表示handle，即外设的完整配置和状态集合。如`hADC`、`hIIC`、`hUSART`。
- **`SPI_GPIO`结构体**：用于同时定义SPI的MOSI、MISO、SCLK三个引脚。
- **`LED`结构体**：专门针对LED这种简单外设的抽象。
- **`rx_info`结构体**：用于描述接收数据的上下文信息。

这套前缀系统的核心思想是**类型即文档**。当我看到`hADC`时，我立即知道这是一个包含ADC外设所有配置信息的结构体实例；当我看到`SPI_GPIO`时，我知道这是与SPI通信相关的GPIO配置集合。

### 2.2 函数命名约定

我的函数命名遵循一个简单的动词-配置模式：

- `xxx_init_config`：初始化某个外设的配置
- `led_on`、`led_off`：状态操作函数，使用动词
- `gd25q_CS_active`、`gd25q_CS_inactive`：片选操作，用`_active`和`_inactive`表示状态

这种命名方式的优势在于**函数名的自文档化**。当你调用`usart_init_config(&USB_UART)`时，你清楚地知道这个函数是在"配置"而非"读取"或"启用"。

### 2.3 变量的命名艺术

变量命名是我花费最多心思的地方。在嵌入式开发中，变量通常代表硬件状态或传感器数据，因此命名必须**精确且有上下文**：

```c
uint8_t uart_buf[100];        // UART缓冲区，名称简短因为类型已经说明这是缓冲区
uint32_t adc_value;           // ADC转换值，直接用物理量命名
float vcc;                    // 电源电压，用物理符号更直观
```

数组初始化时的注释也是我风格的一部分：

```c
uint8_t array[100] = {
  0x00, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00,  // 0~9: 0x00
  0x01, 0x01, 0x01, 0x01, 0x01, 0x01, 0x01, 0x01, 0x01, 0x01,  // 10~19: 0x01
  // ...
};
```

这种注释风格不是为了记录代码意图（代码本身应该自解释），而是为了在调试时快速定位数组索引与物理位置的对应关系。

## 三、结构体的设计哲学

### 3.1 配置结构体

在我的代码中，结构体是组织配置数据的核心工具。以USART配置为例：

```c
typedef struct
{
    uint32_t usart_periph;
    uint32_t tx_gpio_periph;
    uint32_t tx_pin;
    uint32_t rx_gpio_periph;
    uint32_t rx_pin;
    uint32_t baudval;
    uint32_t paritycfg;
    uint32_t wlen;
    uint32_t stblen;
}USART;
```

这个结构体的设计遵循**聚合相关数据**的原则。USART通信涉及的所有配置（外设选择、GPIO引脚、通信参数）都被放在一个结构体里，而不是作为独立的全局变量。这种设计的好处是：

1. **代码可读性**：所有USART相关配置都聚拢在一起
2. **接口简洁**：初始化函数只需接收一个指针
3. **易于扩展**：新增配置项只需修改结构体定义

### 3.2 函数指针结构体（C中的多态）

在gd30ad3344.c中，我展示了一种更高级的结构体用法——用函数指针模拟面向对象的行为：

```c
struct gd30ad3344_t
{
    uint16_t config_register;
    int16_t conversion_data;

    void (*transmit)(uint8_t *data, uint16_t lenth);
    void (*receive)(uint8_t *data, uint16_t lenth);
    void (*transmit_receive)(uint8_t *transmit_data, uint8_t *receive_data, uint16_t lenth);

    void (*start_single_conversion)(gd30ad3344_t *this);
    void (*config_MUX)(gd30ad3344_t *this, MUX_options_t mux_option);
    // ...
};
```

这是C语言中实现"封装"的一种方式。ADC外设的所有操作都通过函数指针调用，而具体实现可以在初始化时绑定。这让我可以在不修改调用代码的情况下，更换底层实现（比如从软件SPI切换到硬件SPI）。

## 四、注释的克制与精准

### 4.1 注释的目的

我的注释风格是**克制而精准**的。我不写解释"做什么"的注释（代码本身应该已经清晰），我只在注释解释"为什么这样做"或者"需要注意什么"：

```c
//Configuration Begin
USB_UART=(USART){USART0, GPIOA, GPIO_PIN_9, GPIOA, GPIO_PIN_10, 115200, USART_PM_NONE, USART_WL_8BIT, USART_STB_1BIT};
ADC_GPIO=(GPIO){GPIOC, GPIO_PIN_0, GPIO_MODE_ANALOG, GPIO_PUPD_PULLDOWN};
hADC0=(hADC){ADC0, ADC_ADCCK_PCLK2_DIV6};
//Configuration Ends
```

这里没有注释，因为赋值的结构已经足够说明意图。

### 4.2 todo注释

我在代码中会使用`//todo:`标记未完成或有待改进的地方：

```c
//todo:封装 **第三个参数不能随便设置，每个io都有一一对应的adc通道**
```

```c
//todo:封装 **timeout**
```

```c
//todo:封装
```

这些todo注释不是技术债的记录，而是**思考过程的快照**。当我发现某个地方可以进一步抽象但当前不是最佳时机时，我会记录下来，以免忘记。

### 4.3 特殊注释

对于可能有歧义的代码，我会添加简短说明：

```c
spi_parameter_struct spi_init_struct={
    SPI_MASTER,//主机
    SPI_TRANSMODE_FULLDUPLEX,//全双工
    SPI_FRAMESIZE_8BIT,//8bits
    SPI_NSS_SOFT,//软件片选
    SPI_ENDIAN_MSB,//高位先行
    SPI_CK_PL_LOW_PH_1EDGE,//采样边沿，根据从机选择。该芯片要求片选后SCLK为低电平，在上升沿采样
    SPI_PSC_64//APB时钟分频
};
```

这里每个配置项后面都跟了一个简短注释，因为这些初始化值需要理解硬件协议才能明白为什么这样选择。

## 五、代码的节奏感

### 5.1 空行的运用

我在代码中会使用空行来创造"呼吸空间"。以usart_init_config函数为例：

```c
void usart_init_config(USART *pUSART)
{
    rcu_periph_enum periph;
    GPIO GPIO_TX;
    GPIO GPIO_RX;
    switch(pUSART->usart_periph)
    {
        case USART0:
            periph=RCU_USART0;
            break;
        // ... 更多case
    }
    rcu_periph_clock_enable(periph);
    
    GPIO_TX.gpio_periph=pUSART->tx_gpio_periph;
    // ... TX配置
    
    GPIO_RX.gpio_periph=pUSART->rx_gpio_periph;
    // ... RX配置
    
    usart_baudrate_set(pUSART->usart_periph, pUSART->baudval);
    usart_parity_config(pUSART->usart_periph, pUSART->paritycfg);
    // ...
    
    nvic_irq_enable(USART0_IRQn, 1, 1);
    usart_interrupt_enable(pUSART->usart_periph, USART_INT_RBNE);
    
    usart_enable(pUSART->usart_periph);
}
```

这段代码在不同逻辑块之间有明确的空行分隔：
- 时钟配置
- TX引脚配置
- RX引脚配置
- USART参数配置
- 中断配置
- 启用外设

这种节奏让阅读者能够自然地识别代码的逻辑段落。

### 5.2 switch语句的风格

我的switch语句遵循一种略微非标准的风格：

```c
switch(pUSART->usart_periph)
{
    case USART0:
        periph=RCU_USART0;
        break;
    case USART1:
        periph=RCU_USART1;
        break;
    // ...
}
```

这里每个case只有一个赋值语句，然后立即break。这种风格的优势是**扁平化**，容易扫描；缺点是对于复杂的case逻辑会显得冗长。我选择这种风格是因为在我的项目中，switch主要用在外设选择器上，case逻辑通常都很简单。

### 5.3 if语句的统一格式

无论分支多么简单，我都使用大括号：

```c
if(usart_interrupt_flag_get(USART0, USART_INT_FLAG_RBNE)==SET)
{
    usart_interrupt_flag_clear(USART0, USART_INT_FLAG_RBNE);
    *(USB_UART_rx_info.buf_address++)=usart_data_receive(USART0);
}
```

这与我看到的很多嵌入式代码风格不同（它们通常会省略单行if的大括号），但我认为**一致性比简洁更重要**。省略大括号会在未来维护时引入风险——如果需要在if块中添加代码，必须记得加回大括号。

## 六、嵌入式特有的考量

### 6.1 状态机的自然表达

在gd25q.c中，我对Flash操作采用了状态机风格的代码组织：

```c
void gd25q_write_enable(gd25q *pgd25q)
{
    gd25q_CS_active(pgd25q);
    spi_send_rec_byte(pgd25q->spi_periph, WRITE_ENABLE);
    gd25q_CS_inactive(pgd25q);
}

void gd25q_wait_for_write_end(gd25q *pgd25q)
{
    uint8_t gd25q_status = 0;
    gd25q_CS_active(pgd25q);
    spi_send_rec_byte(pgd25q->spi_periph, READ_STATR_REGISTER);
    do {
        gd25q_status = spi_send_rec_byte(pgd25q->spi_periph, DUMMY_BYTE);
    } while(SET == (gd25q_status & WIP_FLAG));
    gd25q_CS_inactive(pgd25q);
}
```

每个操作都是一个独立的函数，片选激活和去激活在每个函数中显式管理。这种风格的代价是有一些重复的CS管理代码，但换来的是每个操作的自包含性——调用者不需要关心片选时序。

### 6.2 硬件抽象的尝试

我在项目中尝试了不同程度的硬件抽象。以GPIO初始化为例：

```c
void gpio_init_config(GPIO *pGPIO)
{
    rcu_periph_clock_enable(rcu_periph_add[pGPIO->gpio_periph]);
    gpio_mode_set(pGPIO->gpio_periph, pGPIO->mode, pGPIO->pull_up_down, pGPIO->pin);
    if(pGPIO->mode == GPIO_MODE_AF)
    {
        gpio_af_set(pGPIO->gpio_periph, pGPIO->alt_func_num, pGPIO->pin);
    }
    if(pGPIO->mode == GPIO_MODE_OUTPUT)
    {
        gpio_output_options_set(pGPIO->gpio_periph, pGPIO->otype, pGPIO->ospeed, pGPIO->pin);
    }
}
```

这个函数试图用统一的接口处理不同模式的GPIO初始化。但这种抽象是有代价的——它需要一些条件判断来区分不同模式，而且并不是所有GD32的GPIO配置都能用这种简单的方式抽象。

### 6.3 handle模式的优势与局限

Handle模式是我在嵌入式开发中广泛使用的设计模式：

```c
USART USB_UART;
hADC hADC0;
GPIO ADC_GPIO;
```

这些全局handle变量代表了我的设计哲学：**硬件即数据结构**。与其分散地管理各种硬件参数，不如把它们聚合成结构体，通过句柄来操作。

然而，这种模式也有其局限性：
1. 对于非常简单的外设（如只用一个引脚驱动LED），抽象反而增加了复杂度
2. 函数指针的使用会略微增加运行时代码大小
3. 调试时函数指针的调用链不如直接函数调用直观

我在项目中会根据实际情况选择是否使用handle模式。对于复杂的通信协议（USART、SPI、I2C），handle模式很有价值；对于简单的GPIO操作，直接操作可能更清晰。

## 七、代码风格的一致性与进化

### 7.1 一致性优先

在我所有的项目中，代码风格保持高度一致。这种一致性体现在：
- 区块划分方式
- 命名约定
- 缩进和空格
- 注释密度

这种一致性让我在切换项目时不需要重新适应代码风格，也让我能够快速在不同项目中复用代码。

### 7.2 风格的演进

我的代码风格并非一成不变。随着经验积累，我会不断调整和改进。以main函数的分区为例，这个分区方式是在多个项目迭代后逐渐形成的——每次发现新的需要区分的代码类别时，我就会扩展这个模式。

同样，对于结构体设计、函数命名等方面，我也在不断尝试更好的组织方式。gd30ad3344.c中的函数指针模式就是一个相对较新的尝试，它代表了我对C语言如何实现封装和接口的思考。

### 7.3 风格与效率的平衡

在嵌入式开发中，效率往往与代码美观存在张力。例如：

```c
gpio_bit_write(pgd25q->CS_GPIO_Port, pgd25q->CS_GPIO_Pin, RESET);
```

这里使用了一个长变量名来提高可读性，但在性能关键的代码路径中，这可能会导致略微低效的代码（尽管编译器通常能优化这一点）。

我倾向于**在非关键路径上优先可读性，在关键路径上优先效率**。但这种选择需要根据具体项目的约束来判断。

## 结语

代码风格不仅仅是格式化规则，更是思考方式的体现。在嵌入式开发这个特殊领域，我们需要同时处理硬件抽象、实时约束、资源限制等多个复杂问题。一个好的代码风格应该能够帮助我们：

1. **快速定位问题**：通过一致的代码组织实现
2. **理解代码意图**：通过自文档化的命名和精准的注释
3. **安全地修改代码**：通过减少隐式依赖和明确的作用域

我形成的这套代码风格，是在无数次调试、重构、移植的经历中逐渐磨炼出来的。它不是最优的，也不是固定的——它是一个持续演进的工具，服务于我编写可靠嵌入式软件的目标。

最重要的是，**风格应该是有意识的**。每一处缩进、每一个命名、每一个注释决定，都应该是经过思考的选择，而不是习惯的延续。真正的代码风格不在于遵循多少规则，而在于是否理解每条规则背后的道理。
